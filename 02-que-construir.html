01 · NB01–02

Ingesta + chunking justificado

Carga el documento. Elige estrategia (fixed / recursive / por estructura / semántico) + tamaño + overlap y justifica por qué para ese tipo de doc. Ideal: prueba ≥2 configs. Conserva metadata (fuente, sección).

02 · NB03

Vector store en Qdrant Cloud

Embeddings (text-embedding-3-large, 256d) → índice en Qdrant Cloud (obligatorio, no el FAISS local). Muestra cuántos vectores hay y un registro de ejemplo (id + payload + texto).

demo03 · NB03

RAG básico con fuentes

Pregunta → embed → similarity search top-k → prompt con contexto → responde solo con lo recuperado, citando la fuente. Demuéstralo en ≥3 preguntas.

La base vectorial debe ser Qdrant Cloud (tiene free tier) — no el fallback FAISS local. Guía de cuenta y datos: docs/setup/02-qdrant-cuenta-y-datos.md.

Aquí están los puntos

Nivel 2 — re-ranking y evaluación no simple

  • Re-ranking LLM-as-judge — recupera top-20 → un LLM puntúa la relevancia de cada pasaje → ordena de mayor a menor → filtra por threshold → genera solo con los que pasan
  • Mini-eval comparativa — 3–5 preguntas propias; corre RAG básico vs re-ranked; tabla y conclusión: ¿mejoró?, ¿por qué?

Es el pipeline del Notebook 04 aplicado a tu documento — su rúbrica es eval-prompt.md.

score_relevancia()
Eres un evaluador de relevancia. Pregunta: {pregunta} Pasaje: {chunk} ¿Qué tan relevante es este pasaje para responder la pregunta? Devuelve SOLO un número entre 0 y 1.

El score lo da el LLM, no el coseno. El threshold decide qué pasajes sobreviven para la respuesta final.
